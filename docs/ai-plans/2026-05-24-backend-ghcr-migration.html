Implementation Plan: Backend GHCR Migration

Approval required before implementation

Task Summary

Move only the Jerky Vault backend image publishing from Docker Hub to GitHub Container Registry. Frontend migration and other services are explicitly out of scope for this chat.

Confirmed Scope

  • In scope: D:\Projects\Code\Personal\jerky-vault-back GitHub Actions image publish path.
  • In scope: backend deploy image reference for jerkyvault in personal_ai_stack, if we choose to include deploy config in the same PR/workflow.
  • Out of scope: D:\Projects\Code\Personal\jerky-vault frontend workflow and frontend deploy image.
  • Out of scope: other Docker Hub services, because they have already been moved or are being handled separately.

Current Behavior

  • Backend workflow .github/workflows/main.yml logs into Docker Hub using DOCKER_USERNAME and DOCKER_PASSWORD.
  • Backend workflow pushes ${{ secrets.DOCKER_USERNAME }}/jerky_vault_backend:latest.
  • Deploy config currently references dzarlax/jerky_vault_backend:latest in D:\Projects\Code\Personal\personal_ai_stack\deploy\jerkyvault\docker-compose.yaml.
  • The backend PR branch currently has unrelated review-fix work and an unstaged mode-only change on mobile-backend-go; those should not be mixed into this migration without an explicit decision.

Desired Behavior

  • Backend CI publishes to ghcr.io/dzarlax/jerky-vault-back:latest.
  • CI uses GITHUB_TOKEN with packages: write permission instead of Docker Hub credentials.
  • Prefer also publishing a commit SHA tag, for example ghcr.io/dzarlax/jerky-vault-back:${{ github.sha }}, for rollback/debugging.
  • Deploy config should eventually consume the GHCR image after a successful package is visible and pullable.

Recommended Approach

Use a backend-only PR that changes the backend workflow to GHCR and leaves deploy config as a follow-up until the first GHCR image is produced. This avoids changing production pull targets before the package exists. After merge and successful CI, update personal_ai_stack deploy config in a separate deploy-focused change.

Alternative

Change backend CI and personal_ai_stack deploy image in one coordinated change. This is faster, but riskier: if GHCR package visibility or auth is wrong, the next deploy can fail to pull the backend image.

Files Likely to Change

  • .github/workflows/main.yml - replace Docker Hub login/tagging with GHCR login/tagging.
  • README.md or DOCKER.md if present/needed - document the new image name.
  • docs/ai-plans/2026-05-24-backend-ghcr-migration.html - update after implementation with actual checks and follow-ups.
  • Optional follow-up outside backend repo: personal_ai_stack/deploy/jerkyvault/docker-compose.yaml - backend image reference after GHCR image is confirmed.

Implementation Steps

  1. Start from a clean backend branch based on the intended base branch, not mixed with frontend changes.
  2. Update workflow permissions with contents: read and packages: write.
  3. Change Docker login registry to ghcr.io, username to ${{ github.actor }}, password to ${{ secrets.GITHUB_TOKEN }}.
  4. Change image tags to ghcr.io/dzarlax/jerky-vault-back:latest and ghcr.io/dzarlax/jerky-vault-back:${{ github.sha }}.
  5. Keep Dockerfile build behavior unchanged.
  6. Run local YAML review and, if practical, a local Docker build smoke check.
  7. Open a draft PR for backend-only workflow migration.
  8. After the PR merges and CI publishes the GHCR package, verify package visibility and then plan/deploy the compose image switch.

Test Plan

  • docker build -t jerky-vault-back:ghcr-migration .
  • Inspect workflow syntax manually or with an available YAML/action linter if present.
  • After pushing PR branch, verify GitHub Actions can authenticate to GHCR in PR/merge context where applicable.
  • After merge, confirm package exists at ghcr.io/dzarlax/jerky-vault-back.
  • Before any deploy config switch, verify the VPS can pull the GHCR image or that the package is public.

Risks and Edge Cases

  • GHCR packages can be private by default; VPS pull will fail unless the package is public or the server is logged in.
  • Changing deploy config before the first GHCR image exists can break the next backend restart/deploy.
  • Docker Hub and GHCR image names differ; scripts/docs must use the exact GHCR path.
  • The current backend working tree has an unrelated unstaged binary mode change; keep it out of this PR.

Rollback Plan

If GHCR publishing fails, revert the workflow tags/login back to Docker Hub. If deploy config is later switched and pull fails, revert only the compose image reference to dzarlax/jerky_vault_backend:latest and restart the backend service.
