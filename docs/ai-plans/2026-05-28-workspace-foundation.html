Implementation Plan: Workspace Foundation

This plan covers the first backend-only step toward workspace-scoped operational data in jerky-vault-back.

Date: 2026-05-28 Branch: codex/workspace-foundation-plan Approval required before implementation

Task Summary

Add a workspace foundation without migrating business tables yet. Users stay identities. New workspaces become the future operational boundary. Each existing user gets a default personal workspace and owner membership. Protected requests can resolve a current workspace and verify membership.

This step intentionally stops before moving recipes, prices, products, clients, orders, packages, or cooking_sessions from user_id to workspace_id.

Current Behavior

  • middleware/auth.go validates JWTs and stores only userID in Gin context.
  • Most operational controllers filter by user_id, including recipes, prices, products, clients, orders, packages, cooking sessions, and dashboard queries.
  • models.Ingredient has no user_id. Its name field is globally unique.
  • controllers/ingredients.go returns the full global ingredient list and creates global ingredients.
  • controllers/prices.go stores price history rows by user_id and ingredient_id.
  • Schema setup is currently driven by GORM AutoMigrate plus explicit index creation in database/database.go; database/init.go has a second AutoMigrate list.
  • Existing tests are lightweight unit tests only; no current workspace or controller integration tests exist.

Desired Behavior

  • Add Workspace and WorkspaceMember models and migrate them on startup.
  • Backfill one default personal workspace for every existing user, with an owner membership.
  • On protected requests, resolve the current workspace from X-Workspace-ID when present.
  • If X-Workspace-ID is missing, fall back to the user's default personal workspace to preserve current frontend behavior.
  • Reject requests that specify a workspace the authenticated user is not a member of.
  • Store both userID and workspaceID in Gin context for future controllers.
  • Add read/bootstrap endpoints only, so the frontend can discover available workspaces without introducing workspace creation before business data is workspace-owned.

Scope Boundary

In Scope

  • Workspace and membership data model.
  • Default personal workspace backfill.
  • Workspace resolution middleware/helpers.
  • Membership enforcement tests.
  • Basic workspace listing and current workspace endpoint, including the user's role.
  • Documentation updates where existing instructions mention user-only ownership.

Out of Scope

  • Adding workspace_id to prices or other business tables.
  • Changing ingredient list behavior to workspace ingredients.
  • Adding WorkspaceIngredient.
  • Adding account or company ownership.
  • Frontend workspace selector implementation.
  • Removing or renaming existing user_id columns.
  • Workspace creation and member management endpoints.

Implementation Steps

  1. Add workspace role constants in constants/workspace_roles.go, starting with owner, manager, operator, and viewer.
  2. Add Workspace and WorkspaceMember GORM models with soft delete support and JSON fields.
  3. Resolve the duplicate AutoMigrate lists before adding new models:
    • Delete database/init.go; database.CreateTables() is not called anywhere.
    • Keep database.ConnectDatabase() as the authoritative migration path.
  4. Add explicit indexes:
    • idx_workspace_members_user_id on workspace_members(user_id).
    • idx_workspace_members_workspace_id on workspace_members(workspace_id).
    • Partial unique index on active (workspace_id, user_id).
    • A uniqueness guard for per-user default personal workspaces using nullable internal personal_user_id and a unique partial index where not deleted.
  5. Add idempotent backfill:
    • Find every user.
    • Create a personal workspace if one does not already exist for that user.
    • Create owner membership if absent.
    • Run in a transaction and tolerate repeated startup runs.
  6. Add workspace resolver as separate middleware in middleware/workspace.go and wire it after JWTMiddleware():
    • Parse optional X-Workspace-ID as an unsigned integer.
    • Treat a missing or blank X-Workspace-ID as fallback to the default personal workspace.
    • Return 400 Bad Request for malformed, negative, or zero workspace ids.
    • When present, require active membership for userID.
    • When absent, load or create the default personal workspace for userID.
    • Set workspaceID and optionally workspaceRole in Gin context.
  7. Use database indexes for the added request lookup paths; do not add a workspace cache in this phase. One indexed membership lookup or personal workspace lookup per protected request is acceptable for this migration step.
  8. Add workspace bootstrap endpoints:
    • GET /api/workspaces returns active workspaces the user belongs to, with role.
    • GET /api/workspaces/current returns the resolved current workspace and role.
  9. Add tests for backfill and membership enforcement.
  10. Update documentation to state that workspaceID now exists in request context, while legacy operational data still uses user_id until later phases.

Data, API, Auth, and UX Impact

  • Data: adds two new tables and indexes. Existing rows in operational tables are not changed in this phase.
  • API: existing endpoints should keep working because default workspace fallback resolves a workspace when no header is sent. This phase adds only workspace read/bootstrap endpoints, not creation or member management.
  • Auth: JWT payload remains unchanged; workspace authorization is derived from database membership after token validation.
  • Permissions: this phase validates membership only. Role-specific permissions are stored through constants but not enforced beyond membership.
  • UX: no visible ingredient or pricing behavior changes yet. Frontend can start reading GET /api/workspaces and sending X-Workspace-ID later.
  • Deployment: startup will run AutoMigrate and the workspace backfill. This should be tested on a database copy or staging-equivalent database before production deploy.

  • go test ./... for all unit and new middleware/backfill tests.
  • go vet ./... for static checks.
  • go build ./... to confirm the service compiles.

Manual QA Checklist

  • Register or log in as a user and call an existing endpoint without X-Workspace-ID; it should still work.
  • Call GET /api/workspaces; it should return the user's personal workspace.
  • Call GET /api/workspaces/current without a header; it should return the personal workspace and role.
  • Call a protected endpoint with another user's workspace id; it should return a clear authorization error.
  • Restart the service and verify workspace counts do not increase unexpectedly.
  • Smoke check ingredients and prices endpoints to confirm no behavior changed in this phase.

Risks and Edge Cases

  • AutoMigrate plus backfill can hide migration ordering mistakes. Mitigation: make backfill explicit, idempotent, run after schema creation, and remove or consolidate the duplicate AutoMigrate path before adding models.
  • Default fallback can mask missing workspace selection in future frontend code. Mitigation: log fallback use at debug/info level during migration and make new UI send explicit workspace ids.
  • If personal workspace uniqueness is based only on generated slug, display name changes or slug collisions can create ambiguity. Mitigation: use stable internal personal_user_id with an index, hidden from JSON.
  • Applying workspace middleware to all protected routes may affect endpoints that only need profile-level auth. Mitigation: keep fallback cheap and deterministic; if needed, split profile routes into auth-only later.
  • Documentation drift is likely because CLAUDE.md currently says all data isolation is by user_id. Mitigation: update it in the same implementation pass.
  • Role storage without role enforcement can be misunderstood. Mitigation: document that roles are reserved for later authorization rules.

Rollback Plan

  • Because no existing business table ownership changes in this phase, rollback is mostly code-level: redeploy the previous backend image or commit.
  • Leaving workspaces and workspace_members tables in the database is safe if old code ignores them.
  • If cleanup is required, drop only the new workspace tables and their indexes after confirming no later phase has started writing workspace_id to business tables.
  • Do not remove or mutate existing user_id data during this phase.

Open Questions

  1. Default personal workspace name: use a neutral display value such as Personal workspace, unless product copy requires a localized/frontend-provided label later.
  2. personal_user_id: keep internal with json:"-".
  3. GET /api/workspaces/current: include the user's role.
  4. Workspace creation endpoints: defer. This foundation phase exposes only read/bootstrap endpoints plus backfill.
  5. Role-specific permissions: defer until workspace creation/member management or business-data migration needs them.
