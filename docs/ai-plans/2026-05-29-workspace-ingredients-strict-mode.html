Workspace Ingredients Strict Mode

Final backend step after workspace-aware web and mobile clients have been deployed.

Approval Gate

This is a planning-only change. Do not modify production code until this plan is approved. The intended implementation is small, but it changes write behavior and can break stale clients that still submit global ingredient ids without first linking them to the active workspace.

Recommended approval condition: confirm the web app, mobile app, and widget are all deployed with workspace-aware ingredient pickers and X-Workspace-ID propagation.

  • Verify the deployed web bundle sends X-Workspace-ID through the shared fetcher and uses /api/workspace-ingredients for price and recipe ingredient pickers.
  • Verify the deployed mobile app sends X-Workspace-ID from the shared API client and uses workspace ingredients for price and recipe ingredient pickers.
  • Verify the deployed widget sends the selected workspace id for dashboard and profit requests.
  • Verify stale or missing workspace ids produce validation/client recovery behavior, not auth logout behavior.

Summary

Phase 3A intentionally kept compatibility auto-linking: when a client submitted a valid global ingredient id to price or recipe-ingredient writes, the backend linked that ingredient into the active workspace automatically.

Now that clients are expected to use /api/workspace-ingredients for pickers and explicit linking, strict mode should remove that hidden auto-link from write paths. Price and recipe-ingredient writes should only accept ingredients that are active members of the selected workspace when STRICT_WORKSPACE_INGREDIENTS=true.

The feature flag should default off. This allows a dark deploy of the code and a fast environment rollback while older mobile builds may still exist in the field.

Current Behavior

Endpoint Current behavior Why it existed
POST /api/prices Validates that the global ingredient exists, then calls EnsureWorkspaceIngredient before creating the price. Compatibility for old clients using the global ingredient list as a picker.
POST /api/recipes/:id/ingredients Validates the recipe is in the active workspace, validates the global ingredient exists, then calls EnsureWorkspaceIngredient. Compatibility for old recipe-edit flows.
POST /api/ingredients Creates a global ingredient or returns 409 for an existing name, and ensures workspace membership in both cases. This is an explicit create-or-add flow and should remain valid.
POST /api/workspace-ingredients Links an existing global ingredient to the active workspace through EnsureWorkspaceIngredient. This is the explicit workspace working-set management endpoint and should remain valid.

Desired Behavior

  • With STRICT_WORKSPACE_INGREDIENTS=false or unset, current compatibility auto-link behavior remains active.
  • With STRICT_WORKSPACE_INGREDIENTS=true, POST /api/prices rejects valid global ingredients that are not active members of the active workspace.
  • With STRICT_WORKSPACE_INGREDIENTS=true, POST /api/recipes/:id/ingredients rejects valid global ingredients that are not active members of the recipe workspace.
  • Inactive workspace ingredients are treated as not available for new writes only when strict mode is enabled.
  • Invalid global ingredient ids still return the existing validation error path.
  • POST /api/ingredients remains the create-or-add-to-workspace flow.
  • POST /api/workspace-ingredients remains the explicit link/reactivate flow.
  • GET /api/ingredients and GET /api/ingredients/search remain global dictionary endpoints.
  • Price list reads stay scoped by prices.workspace_id only. Do not hide existing price history because a membership was later deactivated.
  • Recipe reads stay stable for existing recipes. Already-used ingredients must remain visible even if their workspace membership was later deactivated.

Non-Goals

  • No schema change or migration.
  • No backfill change.
  • No change to global ingredient uniqueness or 409 existing_id behavior.
  • No fix for the existing price sort spillover where ingredient_name and ingredient_type are listed as sortable without a joined ingredient query.
  • No frontend or mobile changes in this backend PR.

Implementation Steps

  1. Add strict-mode config parsing, for example STRICT_WORKSPACE_INGREDIENTS=true, defaulting to off when unset.
  2. Add a small strict validation helper, for example RequireWorkspaceIngredient(db, workspaceID, ingredientID), or inline IngredientInWorkspace checks in the two controllers.
  3. Keep the helper active-only: workspace_ingredients.active = true and non-deleted rows only.
  4. In AddPrice, keep the existing global ingredient existence check. If strict mode is on, require active workspace membership. If strict mode is off, keep the existing EnsureWorkspaceIngredient compatibility path.
  5. In AddIngredientToRecipe, keep the recipe workspace ownership check and global ingredient existence check. If strict mode is on, require active workspace membership. If strict mode is off, keep the existing EnsureWorkspaceIngredient compatibility path.
  6. Return a non-auth failure for valid-but-not-linked ingredients. Recommended status: 400, with a clear message such as Ingredient is not in workspace. Avoid 403 because existing clients may treat it as authentication failure.
  7. Leave CreateIngredient and AddWorkspaceIngredient using EnsureWorkspaceIngredient. Those endpoints are explicit add/link flows.
  8. Audit every remaining EnsureWorkspaceIngredient call. Expected controller callers after the change: ingredient create/existing-name path, workspace ingredient add path, and the explicit compatibility branch used only when STRICT_WORKSPACE_INGREDIENTS is off.
  9. Update Swagger annotations if the documented error behavior for price or recipe-ingredient writes changes.
  10. Regenerate Swagger docs if annotations changed.

Test Plan

  • With STRICT_WORKSPACE_INGREDIENTS=false or unset, price write with a valid global but unlinked ingredient keeps old compatibility behavior and auto-links the workspace ingredient.
  • With STRICT_WORKSPACE_INGREDIENTS=true, price write with active workspace ingredient succeeds.
  • With STRICT_WORKSPACE_INGREDIENTS=true, price write with valid global ingredient that is not linked to the workspace returns 400 and does not create a price.
  • With STRICT_WORKSPACE_INGREDIENTS=true, price write with deactivated workspace ingredient returns 400 and does not create a price.
  • With STRICT_WORKSPACE_INGREDIENTS=false or unset, recipe ingredient write with a valid global but unlinked ingredient keeps old compatibility behavior and auto-links the workspace ingredient.
  • With STRICT_WORKSPACE_INGREDIENTS=true, recipe ingredient write with active workspace ingredient succeeds.
  • With STRICT_WORKSPACE_INGREDIENTS=true, recipe ingredient write with valid global ingredient that is not linked to the recipe workspace returns 400 and does not create a recipe ingredient.
  • With STRICT_WORKSPACE_INGREDIENTS=true, recipe ingredient write with deactivated workspace ingredient returns 400 and does not create a recipe ingredient.
  • Invalid ingredient id remains rejected through the existing invalid-id path.
  • POST /api/workspace-ingredients can link or reactivate an ingredient, after which price and recipe writes succeed.
  • POST /api/ingredients create path still creates a global ingredient and links it to the active workspace.
  • POST /api/ingredients existing-name 409 path still ensures or reactivates workspace membership.
  • Existing recipes and price history remain readable when a workspace ingredient is later deactivated.

Deployment gate: deploy the code with STRICT_WORKSPACE_INGREDIENTS off first, observe logs, then enable the flag only after web, mobile, and widget clients that depend on workspace ingredient pickers are live enough for the intended user base.

Production Data Checks

No migration is expected. Still, run read-only checks before deployment to confirm there is no obvious data shape problem.

  • Confirm active workspace ingredient backfill remains complete for current usage sources.
  • Confirm workspace_ingredients has rows for active workspaces that already have prices or recipe ingredients.
  • Do not block deployment solely because old historical prices reference inactive membership; strict mode affects new writes, not history reads.

Risks

Risk Impact Mitigation
Old web or mobile clients still submit global ingredient ids. New price or recipe ingredient writes fail. Deploy strict code with the flag off first. Enable STRICT_WORKSPACE_INGREDIENTS only after rollout verification.
Mobile app rollout lag. Installed older builds may fail write flows after strict deployment. Keep the flag off during the observation window. If failures appear after enabling, disable the flag without redeploying the backend image.
Error status accidentally uses 403. Clients may log the user out instead of showing a validation error. Use 400 for non-member ingredient validation failures.
Inactive ingredient display is confused with write eligibility. Existing recipes may appear broken if reads are over-filtered. Only enforce active membership on new writes. Keep historical reads stable.

Rollback Plan

  • Primary rollback: set STRICT_WORKSPACE_INGREDIENTS=false and restart the backend.
  • Backend image rollback remains available: redeploy the previous backend image or revert the strict-mode PR.
  • No database rollback should be required because this plan has no schema or data migration.
  • If client errors appear after deployment, check API logs for Ingredient is not in workspace responses and confirm which client version produced them.

Open Questions

  1. Feature flag decision: resolved in favor of STRICT_WORKSPACE_INGREDIENTS, default off. This supports dark deploy and a fast environment rollback while old mobile builds may still exist.
  2. Final status code for valid global but non-member ingredients: recommendation is 400, not 403, to avoid auth/logout behavior.
  3. How long should old mobile builds be tolerated? If there is no reliable adoption signal, wait a short observation window before strict deployment.
