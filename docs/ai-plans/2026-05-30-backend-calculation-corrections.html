Implementation Plan: Backend Calculation Corrections

Approval required before implementation

Task Summary

Correct backend calculation behavior for recipe ingredient costs, latest workspace price selection, and order item cost handling. The goal is to remove silent wrong totals while keeping existing workspace-scoped data boundaries intact.

Inspected Current Behavior

Recipe Cost

utils.CalculateIngredientCost converts kg -> g, g -> g, l -> ml, and ml -> ml. All other unit pairs fall through to price / priceQuantity * recipeQuantity.

Recipe Totals

controllers/recipes.go recalculates calculated_cost and total_cost from the latest workspace price for each ingredient. Invalid cost calculations are currently skipped, leaving partial totals.

Latest Price

Recipe and workspace ingredient lookups order prices by date DESC only. Multiple prices with the same business date can produce nondeterministic results.

Order Costs

AddOrder and UpdateOrder treat cost_price == 0 as "not provided" and replace it with product.Cost, even though request validation allows zero.

Desired Behavior

  • Recipe cost conversion supports both directions for mass and volume: kg/g and l/ml.
  • Same-unit prices continue to work for mass, volume, and count-like units.
  • Unsupported cross-dimension pairs, empty units where a conversion is required, and unknown units do not produce silent numbers.
  • Latest price lookups use deterministic ordering, preferably date DESC, created_at DESC, id DESC.
  • Order item cost_price can distinguish "field omitted" from "explicitly set to zero".
  • Dashboard revenue, cost, and profit formulas stay unchanged, but receive corrected stored order item costs.

Assumptions and Unknowns

  • Supported ingredient units are currently implicit in API usage and docs, not centralized in code.
  • Count-like units may include empty string, unit, pcs, or other UI values; exact accepted names need either existing frontend confirmation or a conservative backend allowlist.
  • Changing unsupported unit handling from silent fallback to an error may alter API responses for malformed existing recipe ingredients.
  • No database migration is expected unless we decide to persist normalized units or add direct schema constraints later.

Files Likely to Change

  • utils/calculateIngredientCost.go - replace the fallback calculation with explicit normalization, conversion factors, and validation errors.
  • utils/calculateIngredientCost_test.go - add focused table tests for conversions, decimal quantities, invalid quantities, zero price quantity, and unsupported unit pairs.
  • controllers/recipes.go - use deterministic latest-price ordering and decide how calculation errors should surface or be represented in responses.
  • controllers/ingredients.go - use the same deterministic latest-price ordering for latest_price.
  • controllers/prices.go - align default price list ordering with deterministic latest-price semantics if useful for the UI.
  • controllers/orders.go - change request DTOs for order items so omitted cost_price can be distinguished from explicit zero.
  • controllers/workspace_price_test.go - extend coverage for recipe totals under tied price dates and mixed unit conversions.
  • controllers/workspace_business_test.go - add coverage for explicit zero cost_price and profit output.
  • docs/docs.go, docs/swagger.json, docs/swagger.yaml - regenerate only if request/response annotations or documented model shapes change.
  • docs/ai-plans/2026-05-30-backend-calculation-corrections.html - update after implementation with actual changes and checks.

Implementation Steps

  1. Add table-driven tests that capture the expected calculation contract before editing production code.
  2. Introduce a small unit-normalization helper inside utils or the calculation file: trim spaces, lower-case units, map common aliases only if confirmed by existing usage.
  3. Implement explicit conversion through base units: grams for mass, milliliters for volume, and unit counts for count-like units.
  4. Return errors for zero/negative priceQuantity, invalid recipe quantities, unsupported units, and incompatible dimensions.
  5. Update recipe total calculation to use the new helper and keep behavior explicit when an ingredient cannot be calculated.
  6. Change latest price ordering in all read paths from date DESC to deterministic ordering.
  7. Change order item request structs to use *float64 for cost_price, preserving product cost fallback only when the field is omitted.
  8. Add controller tests for explicit cost_price: 0, tied price dates, and cross-workspace-safe recipe cost calculations.
  9. Run formatting, full tests, vet, and regenerate Swagger if the public request schema changes.
  10. Update this plan with completed changes, checks, known limitations, and follow-up recommendations.

API and Data Impact

  • No planned database schema migration.
  • cost_price remains the same JSON field, but backend semantics change: omitted means fallback to product cost; explicit 0 means store zero.
  • Malformed recipe ingredient units may stop producing silent partial totals. The response behavior needs an implementation choice before coding: fail the recipe request, include a calculation warning field, or keep the ingredient cost at zero while logging.
  • Latest price results can change for ingredients with multiple price rows on the same date; this is intended to make the result stable and newest-row-biased.
  • Workspace authorization and scoping should not change.

Manual QA Checklist

  • Create or inspect a recipe using a price in kg and recipe quantity in g; verify total_cost.
  • Create or inspect the reverse case, price in g and recipe quantity in kg; verify total_cost.
  • Repeat for l/ml.
  • Add two prices for the same ingredient with the same date; verify the latest created row wins consistently in recipe details and workspace ingredients.
  • Create an order item with omitted cost_price; verify it uses product cost.
  • Create an order item with explicit cost_price: 0; verify dashboard profit treats cost as zero.
  • Confirm workspace selection still isolates prices, recipes, orders, and dashboard totals.

Risks and Edge Cases

  • Existing data may contain unit spellings that are not currently known from backend code.
  • Returning hard errors from recipe reads could break screens that previously rendered with partial totals.
  • Float arithmetic remains approximate; this plan does not move monetary values to integer minor units.
  • Using binding:"required" with numeric zero can reject valid zero values in some Gin validator paths; tests should cover explicit zero where the API permits it.
  • Swagger output may change if request structs move from inline anonymous structs to named DTOs for pointer fields.

Rollback Plan

Revert the calculation and controller changes in the implementation PR. Because this plan avoids schema migrations, rollback should be an application-code rollback only. If deterministic latest-price ordering exposes unexpected business behavior, it can be reverted independently from the unit conversion and order cost fixes.

Open Questions

  • Which exact unit names should the backend support beyond kg, g, l, and ml?
  • For unsupported unit pairs during recipe reads, should the endpoint fail, omit only that ingredient's calculated cost, or return a warning field?
  • Should prices with identical date prefer newest created_at, highest id, or should same-date duplicates be blocked later?
  • Should money stay as float64 for now, or should a later separate plan move monetary fields to integer minor units?

Known Limitations

  • Supported convertible units are intentionally conservative: kg, g, l, ml, and pcs. Empty units and matching custom units are treated as count-style units for backward compatibility with existing unitless data.
  • Recipe read handlers still skip individual ingredient calculation errors instead of surfacing warnings or failing the request. This preserves existing response behavior but can hide malformed historical units.
  • Money fields remain float64; integer minor-unit storage is out of scope for this correction.
  • Swagger was not regenerated because no public field names or route annotations changed.

Approval Gate

Implementation must not start until this plan is explicitly approved. After approval, production code changes should stay scoped to calculation correctness, deterministic latest-price selection, order cost semantics, tests, and any required Swagger refresh.
