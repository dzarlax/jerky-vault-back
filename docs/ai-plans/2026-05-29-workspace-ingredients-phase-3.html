Implementation Plan: Workspace Ingredients Phase 3

Approval required before implementation

Task Summary

Phase 1 added workspaces and request workspace resolution. Phase 2A/2B migrated prices and primary business tables to workspace_id. Phase 3 should add the missing ingredient working-set layer: workspace_ingredients.

Recommended scope for this backend PR is Phase 3A: schema, backfill, backend API behavior, and server-side integrity enforcement. The frontend workbench should be a follow-up PR in the frontend repository once this API contract is reviewed.

Current Behavior

Workspace Scoped

  • prices are scoped by workspace_id.
  • recipes, products, packages, clients, orders, and cooking_sessions are scoped by workspace_id.
  • Recipe cost lookup reads latest prices in the active workspace.

Still Global

  • ingredients remain a global dictionary with globally unique name.
  • GET /api/ingredients returns every active ingredient in the installation.
  • POST /api/prices accepts any existing global ingredient_id.
  • POST /api/recipes/:id/ingredients accepts any existing global ingredient_id if the recipe belongs to the workspace.

Desired Behavior

  • ingredients remains the shared dictionary. Do not add workspace_id to ingredients.
  • workspace_ingredients becomes the membership layer between a workspace and shared ingredients.
  • GET /api/ingredients stays backward-compatible in Phase 3A and continues to return the global dictionary until the frontend workbench is ready.
  • A new workspace ingredient endpoint returns only ingredients in the active workspace working set.
  • Global dictionary search remains available for add-to-workspace flows.
  • Creating or reusing an ingredient through POST /api/ingredients guarantees active membership in the current workspace, including the existing-name path.
  • In Phase 3A compatibility mode, price and recipe ingredient writes auto-ensure active membership for valid global ingredients before writing. Strict rejection of non-members is deferred until the frontend switches to scoped workspace pickers.
  • Existing production data remains visible because memberships are backfilled from existing prices, recipes, and cooking sessions.

Frontend Compatibility Findings

The sibling frontend repository jerky-vault currently uses GET /api/ingredients as the picker source for the ingredient page, prices page, recipes page, recipe create modal, and recipe edit modal. The shared fetcher does not send X-Workspace-ID, so the deployed UI still relies on the backend's default workspace fallback.

Because POST /api/prices and POST /api/recipes/:id/ingredients are called directly with ids from the global ingredient list, rejecting global-but-not-yet-linked ingredients in backend-only Phase 3A would break existing price and recipe flows. The backend PR should therefore populate membership on writes for valid global ingredients, while exposing the stricter workspace list API for the Phase 3B frontend migration.

Proposed API Contract

  • GET /api/ingredients - compatibility route. Keep returning the global dictionary in Phase 3A so the existing frontend picker does not lose access to not-yet-linked ingredients.
  • GET /api/workspace-ingredients - explicit workspace working-set endpoint with membership metadata and latest price summary.
  • POST /api/workspace-ingredients - add an existing global ingredient to the active workspace by ingredient_id.
  • PATCH /api/workspace-ingredients/:id - update membership metadata such as active, alias, or category.
  • DELETE /api/workspace-ingredients/:id - soft-delete or deactivate membership, but do not delete the global ingredient.
  • GET /api/ingredients/search?query=... - search the global dictionary for add-to-workspace flows.
  • POST /api/ingredients - create a global ingredient and link it to the active workspace in one transaction. If the name already exists globally, ensure membership for the existing ingredient and return 409 with existing_id plus membership/link metadata. This preserves the existing create-or-reuse frontend flow.

Model and Backfill Design

type WorkspaceIngredient struct {
    ID           uint
    CreatedAt    time.Time
    UpdatedAt    time.Time
    DeletedAt    gorm.DeletedAt
    WorkspaceID  uint
    IngredientID uint
    Active       bool
    Alias        string
    Category     string
}

Backfill memberships from active usage:

  • Any active prices row with non-null workspace_id.
  • Any active recipe_ingredients row whose parent recipe is active and has non-null workspace_id.
  • Any active cooking_session_ingredients row whose parent cooking_session is active and has non-null workspace_id.

Use ON CONFLICT DO NOTHING / equivalent GORM clause so duplicate sources converge into one active membership. The validation should fail only if active usage references an ingredient that still has no active workspace membership after backfill.

EnsureWorkspaceIngredient must handle previously inactive or soft-deleted membership rows by reactivating the existing relation instead of creating duplicate logical membership.

Implementation Steps

  1. Add WorkspaceIngredient model and relations.
  2. Add indexes: workspace_id, ingredient_id, (workspace_id, active), and active unique (workspace_id, ingredient_id).
  3. Add backfill from prices, recipe ingredients, and cooking session ingredients.
  4. Add shared helper: EnsureWorkspaceIngredient and IngredientInWorkspace.
  5. Keep GET /api/ingredients global in Phase 3A for frontend compatibility.
  6. Add explicit workspace ingredient endpoints and global search endpoint.
  7. Change POST /api/ingredients to create dictionary row plus membership in one transaction, and to ensure membership for the existing-name conflict path.
  8. In POST /api/prices and POST /api/recipes/:id/ingredients, call EnsureWorkspaceIngredient for valid global ingredient ids before writing. Keep invalid ingredient ids rejected.
  9. Keep GET /api/prices list behavior unchanged except for existing workspace filtering. Do not add a workspace ingredient join to price list reads in Phase 3A.
  10. Keep recipe cost lookup behavior unchanged. Phase 3A write paths should populate membership for compatibility; Phase 3B can switch to strict non-member rejection after the frontend uses scoped pickers.
  11. Regenerate Swagger docs and update this plan with checks run.

Risks and Edge Cases

  • Frontend compatibility: GET /api/ingredients intentionally stays global in Phase 3A. The flip to workspace-scoped default behavior belongs with the frontend workbench in Phase 3B.
  • Write compatibility: current frontend price and recipe forms post ids from the global dictionary directly. Phase 3A must auto-ensure workspace membership on those writes; strict non-member rejection belongs after the Phase 3B frontend switch.
  • Duplicate global names: ingredients.name is globally unique. Phase 3 should not solve aliases or cross-domain naming conflicts unless required for membership metadata.
  • Backfill completeness: missing memberships would make existing usage absent from /api/workspace-ingredients. Backfill validation and manual DB checks are mandatory.
  • Deactivate semantics: deleting a workspace ingredient should not break historical recipes, prices, or sessions. Recommendation: soft-delete/deactivate membership for default list visibility, not hard-delete usage.
  • Stale price definition: stale/current price state needs a threshold. Recommendation: do not implement stale filtering in Phase 3A unless a fixed threshold is explicitly approved.
  • Sort bug spillover: prices.go currently allows ingredient_name/ingredient_type sort values without joining ingredients. Phase 3 touches ingredient joins, so decide whether to fix it in-scope or keep it separate.

Rollback Plan

  • workspace_ingredients is additive. Old code can ignore the table.
  • If deployment fails during backfill validation, revert the image. The new table can remain in the database.
  • If scoped ingredient endpoints regress the UI, keep GET /api/ingredients global and keep write auto-linking enabled while membership data remains populated.
  • Do not drop or rewrite global ingredients rows in Phase 3.

Open Questions

  • Scope split: should Phase 3A be backend/API only and Phase 3B be frontend workbench? Recommendation: yes.
  • Response shape: resolved for Phase 3A: keep /api/ingredients compatibility and add richer /api/workspace-ingredients.
  • Membership delete: should delete mean soft-delete or active=false? Recommendation: active=false for user intent, soft-delete only for true removal/audit cleanup.
  • Stale price: should stale use a fixed threshold, workspace setting, or be deferred? Recommendation: defer stale until frontend workbench requirements are explicit.
  • Write auto-link: resolved for Phase 3A: POST /api/prices and POST /api/recipes/:id/ingredients auto-ensure membership for valid global ingredients. Strict non-member rejection is deferred until the frontend uses /api/workspace-ingredients.

Frontend Follow-Up

The frontend workbench should be planned in the sibling jerky-vault repository after this API plan is approved. After that switch, backend write paths can move from compatibility auto-linking to strict non-member rejection.
