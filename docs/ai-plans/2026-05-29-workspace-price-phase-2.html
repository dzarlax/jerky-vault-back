Implementation Plan: Workspace Price Migration Phase 2

This plan covers the next safe backend slice after the merged workspace foundation PR.

Date: 2026-05-29 Branch: codex/workspace-price-phase-2-plan Latest PR reviewed: #4, merged 2026-05-29 Approval required before implementation

Task Summary

Phase 2 in Todoist is described as migrating business tables to workspace_id. The freshest merged PR, #4, recommends starting that migration with prices, then recipe cost paths, before moving the rest of the operational tables. This plan intentionally scopes the next implementation PR to that first executable slice: add and backfill prices.workspace_id, read and write prices by current workspace, and update recipe cost calculations to use workspace-scoped latest prices.

Recipes, products, packages, clients, orders, and cooking sessions remain on user_id in this slice. They should be migrated in later Phase 2 slices after price behavior is verified.

Current Behavior

  • Protected requests now have userID, workspaceID, workspaceRole, and workspace in Gin context.
  • models.Price has user_id and ingredient_id, but no workspace_id.
  • controllers/prices.go creates and lists prices with user_id filtering.
  • controllers/recipes.go loads latest ingredient prices by ingredient_id and user_id for recipe total cost calculation.
  • ingredients remain a shared global dictionary, and GET /api/ingredients still returns all ingredients.
  • database.ConnectDatabase() uses GORM AutoMigrate plus explicit index creation and startup backfills.

Desired Behavior

  • Add nullable prices.workspace_id while keeping prices.user_id for rollback and audit.
  • Backfill each existing price row to the personal workspace for its current user_id.
  • New price writes set both workspace_id from request context and user_id as the creating identity.
  • Price reads use workspace_id as the primary data boundary.
  • Recipe cost calculation uses the latest price in the current workspace.
  • Existing clients that do not send X-Workspace-ID continue to work through default personal workspace fallback.
  • Do not add WorkspaceIngredient yet. Price creation may reference any valid global ingredient until Phase 3 adds membership enforcement.

Scope Boundary

In Scope

  • Price.WorkspaceID model field and migration.
  • Backfill existing prices from user_id to personal workspace.
  • Workspace-scoped price list and create behavior.
  • Workspace-scoped latest price lookup in recipe responses.
  • Indexes for workspace price lookup.
  • Focused tests around migration and workspace isolation.
  • Docs and Swagger updates for price workspace behavior.

Out of Scope

  • Adding workspace_id to recipes, products, packages, clients, orders, or cooking sessions.
  • Changing global ingredient list behavior.
  • Adding workspace_ingredients or ingredient membership enforcement.
  • Dropping or renaming prices.user_id.
  • Adding workspace creation or member-management endpoints.
  • Frontend workspace selector changes.

Proposed Implementation Steps

  1. Add WorkspaceID *uint to models.Price, plus a workspace relation if useful for preload consistency.
  2. Run AutoMigrate through the existing startup migration path so prices.workspace_id is added as nullable.
  3. Add indexes:
    • idx_prices_workspace_id on prices(workspace_id).
    • idx_prices_workspace_ingredient_date on prices(workspace_id, ingredient_id, date DESC).
    • Keep existing ingredient_id and date indexes during the transition.
  4. Add an idempotent backfill:
    • Find prices where workspace_id IS NULL.
    • Map each row directly through workspaces.personal_user_id = prices.user_id. Do not choose through generic workspace membership, because a user can belong to non-personal workspaces too.
    • Update prices.workspace_id only when a matching personal workspace exists.
    • Fail startup if any legacy price remains without a workspace after backfill, because price reads will switch to workspace filtering.
  5. Update AddPrice:
    • Require workspaceID from context.
    • Set WorkspaceID to the current workspace.
    • Keep UserID as the creator identity.
    • Continue validating that ingredient_id exists globally.
  6. Update GetPrices to filter by current workspace_id instead of user_id. Keep existing query parameters and sorting behavior unless a test exposes a bug.
  7. Update recipe cost lookups in GetRecipes and GetRecipe to fetch latest price by ingredient_id plus current workspace_id.
  8. Keep recipe ownership on user_id in this slice. The expected Phase 2A behavior is mixed but intentional: a user sees only their own recipes, while recipe cost uses the active workspace's price history.
  9. Add focused tests for workspace isolation:
    • Two workspaces can have different prices for the same ingredient.
    • GET /api/prices only returns prices for the selected workspace.
    • Recipe total cost changes when the same recipe is viewed under a workspace with a different latest ingredient price.
    • When the active workspace has no price for a recipe ingredient, the current behavior remains explicit in tests: no latest price is attached and the calculated ingredient cost stays zero.
    • Missing X-Workspace-ID still uses the personal workspace fallback.
  10. Update docs and generated Swagger artifacts after implementation.

Data, API, Auth, and Deployment Impact

  • Data: adds one nullable column to prices and backfills existing rows. No data is deleted.
  • API: GET /api/prices and POST /api/prices keep the same paths and request shapes. Behavior changes from user-scoped to workspace-scoped.
  • Auth: no JWT changes. Workspace membership is already validated by WorkspaceMiddleware().
  • Permissions: no role-specific write restriction in this slice; any member who could call the endpoint before can still call it.
  • Audit semantics: prices.user_id remains in place and should be documented as the creating user field for Phase 2A, not the price ownership boundary.
  • Deployment: startup migration must be verified against a non-production schema first. Backfill should be repeatable and fail closed if legacy price rows cannot be mapped.
  • Rollback: old code can ignore the added nullable column. Keeping user_id preserves the previous boundary if rollback is needed.

Risks and Edge Cases

  • Mixed boundary risk: prices move to workspace_id while recipes still belong to user_id. Mitigation: only change recipe price lookup, not recipe ownership, and test the cost path explicitly.
  • Backfill risk: a legacy price row with a missing or deleted user cannot map to a personal workspace. Mitigation: fail startup in the isolated verification environment and inspect rows before production deploy.
  • Nil workspace risk: nullable column is needed for safe migration, but application reads should not tolerate null after backfill. Mitigation: use startup verification and tests before filtering by workspace.
  • Mixed recipe boundary risk: recipes remain scoped by user_id, but cost lookup uses workspace_id. This is expected for Phase 2A, but it means selecting a different workspace can change costs without changing the recipe list.
  • Global ingredient risk: any workspace can still price any global ingredient until Phase 3. Mitigation: document this as intentional transitional behavior.
  • Sort risk: ingredient_name and ingredient_type are currently listed as valid sort columns but the price query does not join ingredients for SQL ordering. Mitigation: avoid broad sort refactor unless tests show this path is broken in current behavior.

Rollback Plan

  • Redeploy the previous backend image if the price migration code misbehaves.
  • Leave prices.workspace_id in place during rollback; old code ignores it.
  • Do not drop prices.user_id or existing user indexes in this slice.
  • If a later cleanup is required, drop only the new price workspace indexes and column after confirming no newer branch depends on them.

Open Questions

  1. Should the next implementation PR be named and described as Phase 2A to avoid implying that all business tables are migrated at once?
  2. Should startup fail when a price cannot be backfilled, or should it log and keep those rows invisible until manual repair? This plan recommends fail-closed during deployment verification. With direct personal_user_id mapping, the expected unmapped edge is an orphaned price whose user no longer has a resolvable personal workspace.
  3. Should price creation auto-add a future workspace_ingredients membership once Phase 3 exists, or should membership stay an explicit frontend action?

Decisions

  • Name the next implementation PR as Phase 2A, because it migrates prices and recipe cost lookups only.
  • Document prices.user_id as creator/audit semantics during Phase 2A. Do not rename the database column in this slice.
