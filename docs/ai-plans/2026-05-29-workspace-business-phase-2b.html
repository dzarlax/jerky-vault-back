Implementation Plan: Workspace Business Tables Phase 2B

Approval required before implementation

Task Summary

Phase 2A migrated prices and recipe cost lookup to workspace scope. Phase 2B should migrate the remaining primary business ownership tables to workspace_id while keeping user_id as creator/audit data for rollback and compatibility.

Proposed scope: recipes, packages, products, clients, orders, and cooking_sessions. Detail tables such as recipe_ingredients, product_options, order_items, and cooking_session_ingredients should continue to inherit workspace boundaries through their parent records in this phase.

Current Behavior

Already Workspace Scoped

  • prices.workspace_id exists and is backfilled through workspaces.personal_user_id = prices.user_id.
  • POST /api/prices writes the active request workspace.
  • GET /api/prices filters by active workspace.
  • Recipe cost lookup reads latest prices by workspace_id.

Still User Scoped

  • recipes, packages, products, clients, orders, and cooking_sessions models have user_id but no workspace_id.
  • Controllers filter these tables with user_id.
  • Dashboard totals and profit queries filter by orders.user_id.
  • CreateCookingSession currently accepts a recipe_id without validating recipe ownership.

Desired Behavior

  • All primary business records created after this change receive workspace_id from WorkspaceMiddleware.
  • List, detail, update, delete, dashboard, and profit reads use the active workspace as the authorization boundary.
  • user_id remains populated as creator/audit data. It is not renamed in this phase.
  • Cross-workspace references are rejected fail-closed: product package, product recipe options, order client, order products, recipe ingredient parent recipe, and cooking session recipe must belong to the active workspace.
  • Join/detail tables do not get direct workspace_id yet; they inherit it through validated parents.

Files Likely to Change

  • models/recipe.go, models/package.go, models/product.go, models/client.go, models/order.go, models/cooking_session.go - add nullable WorkspaceID and workspace relation.
  • database/business_workspaces.go - add fail-closed backfill for the six parent business tables.
  • database/database.go - run the new backfill after workspace creation and add workspace indexes.
  • controllers/recipes.go, controllers/packages.go, controllers/products.go, controllers/clients.go, controllers/orders.go, controllers/cooking_sessions.go, controllers/recipe_ingredients.go, controllers/dashboard.go - replace ownership filters and reference validation with workspace scoped checks.
  • controllers/*_test.go - add behavioral workspace isolation tests using the existing in-memory SQLite test pattern from Phase 2A.
  • docs/ai-plans/2026-05-29-workspace-business-phase-2b.html - update after implementation with actual checks and known limitations.

Implementation Steps

  1. Add nullable workspace_id and Workspace relations to the six parent business models. Keep user_id required where it is currently required.
  2. Add a shared backfill helper that maps rows through workspaces.personal_user_id = table.user_id and validates that no active rows remain unmapped after backfill.
  3. Add workspace indexes for the migrated tables while keeping existing user_id indexes for rollback and older query paths.
  4. Update create paths to set both UserID and WorkspaceID.
  5. Update read/update/delete filters from user_id ownership to active workspace_id ownership.
  6. Update cross-reference validation: package, recipe, client, product, and order lookups must include active workspace_id.
  7. Update dashboard and profit aggregation queries to filter by workspace_id.
  8. Regenerate tracked Swagger artifacts if handler/model annotations change. This repository tracks docs/docs.go, docs/swagger.json, and docs/swagger.yaml.
  9. Add behavioral tests that prove same-user multi-workspace isolation and cross-workspace rejection.
  10. Run local verification, update this plan with actual checks, then commit, push, and open a PR named as Phase 2B.

Per-Controller Audit Checklist

Because this PR touches several ownership paths, every remaining Where("user_id or equivalent user_id = ? in migrated controllers must be resolved before commit. Each occurrence must either be changed to active workspace_id or explicitly documented as creator/audit logic that should still use user_id.

  • controllers/recipes.go - list, detail, create, delete, and cost lookup parent filtering.
  • controllers/packages.go - list and create ownership boundary.
  • controllers/products.go - list/detail/create/update/delete, package validation, and recipe option validation.
  • controllers/clients.go - list/detail/create/update/delete ownership boundary.
  • controllers/orders.go - list/detail/create/update/status/delete, client validation, product validation, and order item writes.
  • controllers/cooking_sessions.go - list/create ownership boundary and recipe validation before session creation.
  • controllers/recipe_ingredients.go - add/delete ingredient routes must validate the parent recipe in the active workspace before mutating detail rows.
  • controllers/dashboard.go - totals, pending count, recent orders, distribution, and profit must filter by orders.workspace_id or the relevant parent table workspace_id.

Verification command before commit: rg -n 'user_id|UserID|MustGet\("userID"\)' controllers models database -g "*.go", followed by a manual pass over every migrated controller hit.

Backfill Design

Use the same direct personal workspace mapping that Phase 2A uses for prices. Do not backfill through generic workspace_members, because users can belong to non-personal workspaces. 

UPDATE recipes r
SET workspace_id = w.id
FROM workspaces w
WHERE r.workspace_id IS NULL
  AND w.personal_user_id = r.user_id
  AND w.deleted_at IS NULL;

Repeat the pattern for packages, products, clients, orders, and cooking_sessions. The verification query should fail startup only for active rows with non-deleted users that still have workspace_id IS NULL.

Test Plan

  • go test ./... - full Go test suite.
  • go vet ./... - static sanity check.
  • go build ./... - compile all packages.
  • Backfill string tests: each migrated table uses workspaces.personal_user_id = table.user_id, not membership joins.
  • Controller tests: same user, two workspaces, same logical record type; each API returns only active workspace data.
  • Cross-reference tests: product rejects package or recipe from another workspace; order rejects client or product from another workspace; cooking session rejects recipe from another workspace; recipe ingredient routes reject a recipe from another workspace.
  • Cooking-session behavior test: CreateCookingSession must no longer accept a recipe from outside the active workspace. This is a real behavior change, not just a filter swap.
  • Dashboard tests: totals, recent orders, distribution, and profit use only active workspace orders.
  • Audit test/check: no unresolved migrated-controller user_id filters remain except creator/audit assignments that are intentionally preserved.
  • Manual isolated PostgreSQL startup test: run migrations/backfill on a copied or disposable database and verify active unmapped row counts are zero for all migrated tables.

Manual QA Checklist

  • Before switching API filters, seed or verify pre-existing rows with only user_id; after backfill, confirm they remain visible in the user's personal workspace.
  • Create data in personal workspace and a second workspace for the same user.
  • Switch X-Workspace-ID and verify recipes, packages, products, clients, orders, sessions, dashboard, and profit values change accordingly.
  • Verify requests without X-Workspace-ID still resolve to the personal workspace through existing middleware fallback.
  • Attempt cross-workspace IDs in create/update payloads and confirm they are rejected with 400 or 404 rather than silently linking data.
  • For dashboard/profit, verify aggregation filters are applied through parent business tables, especially orders.workspace_id, not through inherited detail rows such as order_items.
  • Confirm production startup logs show migrations, indexes, and backfills completed.

Risks and Edge Cases

  • The broadest risk is mixed boundaries: if one controller path remains on user_id, it can leak same-user data across workspaces. The implementation should audit all controllers that touch migrated tables before commit.
  • Archived rows can have historical invalid references. Backfill validation should fail only on active rows with non-deleted users, matching the Phase 2A production fix.
  • Join/detail rows without direct workspace_id rely on parent validation. Every route that creates or deletes detail records must first verify the parent in the active workspace.
  • CreateCookingSession currently lacks recipe ownership validation. Phase 2B should fix that while switching it to workspace validation.
  • Dashboard is the most fragile query surface because it combines counts, joins, grouping, and aggregate revenue/cost calculations. Tests should cover both dashboard and profit endpoints.
  • The public JSON shape will expose workspace_id for newly migrated models unless explicitly hidden. That is acceptable for API clients but should be treated as an API shape change.

Rollback Plan

  • Keep user_id on all migrated tables and keep existing user_id indexes.
  • If deployment fails during startup backfill, revert the application image to the previous Phase 2A build. Nullable columns created by AutoMigrate are backward-compatible with the old code.
  • If a controller regression is found after deploy, revert the PR and redeploy. Existing workspace_id values can remain in the database without affecting old code.
  • Do not drop old columns or indexes in Phase 2B.

Known Limitations

  • Production database backfill and live API smoke are still pending and should happen only after PR merge/deploy.

Open Questions

  • Scope: include all six parent business tables in one Phase 2B PR, or split further? Recommendation: include all six because orders, products, clients, recipes, and dashboard references are coupled, but require the per-controller audit checklist above before commit.
  • Detail tables: add direct workspace_id to product_options, order_items, recipe_ingredients, and cooking_session_ingredients? Recommendation: no for Phase 2B; enforce via parent and keep schema narrower.
  • Creator semantics: document user_id as creator/audit data for every migrated model? Recommendation: yes, no rename in this phase.
  • Swagger refresh: regenerate checked-in Swagger docs in the same PR when handler/model annotations change, because this repository tracks docs/docs.go, docs/swagger.json, and docs/swagger.yaml.
