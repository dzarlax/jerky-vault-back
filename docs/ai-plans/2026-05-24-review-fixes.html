Current Behavior

Auth

Register returns models.User, and models.User.Password is serialized as password.

Env Loading

middleware/auth.go validates JWT_SECRET in package init(), before main.go loads .env.

Ownership

UpdateOrder accepts a new client_id without validating ownership. Product recipe options accept raw recipe_ids.

Recipe Cost

Recipe cost lookup loads latest price by ingredient_id only, ignoring the authenticated user.

Status Updates

UpdateOrderStatus does not inspect RowsAffected, so a no-op update can still return 200.

Dependencies

govulncheck reported reachable vulnerabilities in the Go standard library path and golang.org/x/net@v0.29.0.

Web Auth Client

jerky-vault/src/pages/auth/signin.tsx only requires token from login and falls back to { username } when data.user is missing.

Mobile Auth Client

Jerky Vault/APIService.swift decodes login as LoginResponse { token }; register decodes User, where password is already optional.

  • Web password change: D:\Projects\Code\Personal\jerky-vault\src\pages\profile.tsx currently enforces minimum 3 characters, and locale text says "Minimum 3 characters". If backend moves to 8, the frontend should be updated in the same pass or the backend should keep 3 intentionally.

Desired Behavior

  • Registration returns only safe public user fields, never password hashes.
  • JWT_SECRET is validated after normal environment loading, including local .env loading.
  • All cross-resource references that should be user-owned are validated before writes.
  • Recipe cost calculations use the authenticated user's price history only.
  • Status update returns 404 when no authenticated user's order was updated.
  • Password minimum length logic matches the returned message.
  • Dependency/runtime updates remove reachable govulncheck findings where possible.

API, Auth, and Data Impact

  • API response shape for POST /api/auth/register will change by removing password.
  • Verified web and mobile login flows do not require a user object from login beyond the token. Mobile register should tolerate missing password because the field is optional.
  • Web signup may currently be broken independently because it calls /api/auth/signup instead of backend /api/auth/register.
  • Invalid or cross-user resource references will return 400 or 404 instead of creating inconsistent rows.
  • Invalid order statuses should return 400 rather than being persisted as arbitrary strings.
  • No planned database schema changes. Existing bad data, if present, will not be automatically repaired in this task.
  • Deployment image build may change if the Docker Go version is pinned to a newer patch.

  • Try updating an order to a foreign or nonexistent client_id; expect rejection.

  • Confirm valid order create/update/status flows still work with known statuses.

Risks and Edge Cases

  • Changing register response is low risk for the inspected clients: mobile User.password is optional and web login does not depend on register response. Any uninspected client could still depend on the old field.
  • Fixing password minimum to 8 in backend without updating the web profile form would create a client/server validation mismatch because the web app currently says 3 characters.
  • Fixing web signup endpoint may touch the neighboring frontend repo, which should be treated as a separate working tree with its own status and validation.
  • Status validation can reject legacy arbitrary statuses if they exist in the database or UI.
  • Dependency updates may require small code adjustments, especially around Gin/GORM transitive dependencies.
  • Docker builder patch availability depends on upstream image tags at implementation time.
  • Tests may need a lightweight test DB strategy; if the project has no established test DB harness, keep test scope focused and document gaps.
